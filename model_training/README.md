# vehiclepm — Model Training (v3)

Three scripts. Run in order.

## What's New in v3

### The "stuck at 1 day" fix
In v2, wear accumulation was **monotonically degrading** — once a component
hit its threshold (from real wear or a fault reading in a CSV), `days_to_service`
would stay at `1` forever. Even after an oil change or brake replacement, the
model had no way to know about it.

**v3 adds service log support.** Drop a CSV (or JSON) in `data/service_logs/`
for each car. When the wear accumulator reaches that date, it resets the relevant
components and the predicted days correctly recover.

### Key changes
| | v2 | v3 |
|---|---|---|
| Wear after service | Stuck at threshold | Resets from service log |
| Model feature count | 33 | 34 (adds `service_applied`) |
| Base model | No service patterns | ~30% of vehicles get a mid-sequence service |
| Output CSV | — | Adds `service_applied`, `services_done`, `drive_date` columns |

## Service Log Format

Place in `data/service_logs/` named `<CarName>_service.csv`
(spaces replaced with underscores, same as the car's folder in `models/finetuned/`).

```csv
service_date,service_type,brake_thickness_after,tire_tread_after,oil_degradation_after,battery_health_after,notes
2026-01-15,oil_change,,,0.05,,Regular oil change
2026-02-20,brake_replacement,9.0,,,,New brake pads
2026-03-10,full_service,9.0,7.5,0.05,,Annual dealer service
2026-04-01,tire_replacement,,7.5,,,Two front tires replaced
2026-04-15,battery_replacement,,,,0.92,New battery
```

### `service_type` values and their defaults

| service_type | What resets | Default value |
|---|---|---|
| `oil_change` | `oil_degradation` | → 0.05 |
| `brake_replacement` | `brake_thickness` | → 9.0 mm |
| `tire_replacement` | `tire_tread` | → 7.5 mm |
| `tire_rotation` | `tire_tread` | → 6.0 mm (partial) |
| `battery_replacement` | `battery_health` | → 0.92 |
| `full_service` | brakes + tires + oil | → 9.0 / 7.5 / 0.05 |
| `custom` | whatever `_after` columns you fill | — |

The `_after` columns **always override defaults**. Leave them blank to use defaults.

### JSON format (alternative)

```json
[
  {"date": "2026-01-15", "type": "oil_change"},
  {"date": "2026-02-20", "type": "brake_replacement", "brake_thickness": 9.5},
  {"date": "2026-03-10", "type": "full_service", "notes": "Tata dealer annual"}
]
```

### Fleet-wide log (one file for all cars)

```csv
vehicle_name,service_date,service_type,brake_thickness_after,...,notes
Safari Storme EX,2026-01-15,oil_change,,,,,Regular change
Nexon EV,2026-02-01,full_service,9.0,7.5,0.05,,Annual
```

## Drive CSV Naming Convention

**Always prefix drive CSV filenames with the drive date:**
```
2026-03-21_session1_Safari_Storme_EX.csv
2026-03-22_session2_Safari_Storme_EX.csv
```

The scripts extract the date from the filename prefix (`YYYY-MM-DD`) to correctly
place service events between the right drives chronologically.
Without date prefixes, services can only be matched by drive index.

## Step 1 — Train Synthetic Base Model

```bash
python train_synthetic_v3.py
```

Generates 5,600 synthetic drive samples. ~30% of vehicles now receive a
mid-sequence service event (oil change, brake replacement, or full service),
giving the base model exposure to post-service recovery patterns.

**Must re-run** if upgrading from v2: the feature count changed (33 → 34)
and fine-tuned models will mismatch.

## Step 2 — Fine-Tune Per Car

```bash
# List cars (shows 📋 icon if a service log exists)
python finetune_car_v3.py --list

# Fine-tune one car
python finetune_car_v3.py --car "Safari Storme EX"

# Fine-tune all cars
python finetune_car_v3.py --all
```

If a service log exists, wear resets are applied at the correct drive.
The `drive_summary.csv` output includes `service_applied` and `services_done`
columns for inspection.

## Step 3 — Predict

```bash
python predict_service_v3.py                        # all cars
python predict_service_v3.py --car "Safari Storme EX"
python predict_service_v3.py --file data/drives/my_drive.csv
python predict_service_v3.py --eval                 # show metrics vs proxy labels
```

The output table now shows a 🔧 icon next to drives where a service was applied.
If the final wear is still past threshold and no service log exists, the script
prints an actionable warning explaining what to add to the service log.

## Troubleshooting

**"Car still shows 1 day after service"**
→ Add an entry to `data/service_logs/<CarName>_service.csv` with the correct
  `service_date` and `service_type`. The date must fall between two drive CSV dates.

**"⚠️ Initial brake_thickness already at/past threshold"**
→ The first drive CSV already shows a worn component. If the car was serviced
  before these drives started, add a service log entry dated *before* the
  first drive date.

**"service_log_utils not found"**
→ Make sure `service_log_utils.py` is in the same directory as the three scripts.

**Feature mismatch / crash after upgrading from v2**
→ Re-run `train_synthetic_v3.py` and then `finetune_car_v3.py --all`.
  v3 has 34 features; v2 models had 33.
